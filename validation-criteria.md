# Develop comprehensive validation criteria

Validation Criteria Development: Create comprehensive validation criteria for all work in this PR. 1) List every change by category (feature, bugfix, refactor, config, test, docs). 2) Per change: define 'done'—what observable behavior proves it works? Be specific: exact inputs, expected outputs, state changes. 3) Happy path: step-by-step verification with concrete test data. 4) Edge cases: empty, null, boundary values, invalid input, concurrent access, partial failure. 5) Integration points: what upstream/downstream systems are affected? How to verify they still work? 6) Regression check: what existing behavior must NOT change? List specific scenarios. 7) Performance criteria: latency, throughput, memory—what's acceptable? 8) Security validation: auth, input sanitization, data exposure—what to verify? 9) Rollback criteria: what signals 'revert immediately'? 10) Manual vs automated: which criteria need human verification? 11) Environment requirements: specific setup, data, or config needed to validate? Rate confidence that criteria are complete—what might be missing (1-3=guessing, 4-6=informed but unverified, 7-8=verified against code/spec, 9-10=verified with stakeholder sign-off)? If any area is too ambiguous to specify criteria (unclear acceptance, missing requirement), STOP and ask one clarifying question rather than inventing criteria. Output as checklist grouped by priority (P0: blocks release, P1: should verify, P2: nice to verify).
